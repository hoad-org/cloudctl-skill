# CloudCTL Skill — MCP Integration Guide

## Purpose

This is a Model Context Protocol (MCP) server that exposes the `cloudctl` binary to Claude Code. It allows Claude to:
- Switch between cloud contexts (AWS, GCP, Azure)
- Execute operations in the correct account/project/subscription
- Manage multi-cloud environments safely

## Architecture

**MCP Server Structure**:
```
┌────────────────────────────────┐
│      Claude Code (mcp client)  │
├────────────────────────────────┤
│  MCP Protocol (JSON-RPC)       │
├────────────────────────────────┤
│  CloudCTL Skill (MCP server)   │
├────────────────────────────────┤
│  cloudctl binary (v4.1.0)      │
├────────────────────────────────┤
│  Cloud Providers (AWS, GCP, Azure)
└────────────────────────────────┘
```

## Integration Points

### From Claude
Claude calls this skill to:
```python
# Switch context
result = await mcp_tools.cloudctl({
    "command": "switch",
    "org": "gcp-terrorgems"
})
# Returns: {account, role, region, exports}

# Get current context
result = await mcp_tools.cloudctl({
    "command": "env"
})
# Returns: {org, account, role, region, provider}
```

### To CloudCTL
The skill wraps the `cloudctl` binary and routes:
- CLI commands to the Python implementation
- Results back to Claude as structured JSON
- Errors handled with appropriate fallbacks

## Key Features

### 1. Provider Routing
The skill works with all three providers:
- **AWS**: IAM Identity Center (SSO)
- **GCP**: gcloud auth + Application Default Credentials
- **Azure**: Azure CLI (az) authentication

### 2. Non-Interactive Support
Handles subprocess contexts (like Claude) gracefully:
- GCP: Falls back to auth confirmation when token refresh fails
- Azure: Uses cached credentials
- AWS: Delegates to SSO cache

### 3. Error Handling
Structured error responses:
```json
{
  "status": "error",
  "code": "NOT_AUTHENTICATED",
  "message": "No active session for org 'gcp-prod'",
  "hint": "Run 'cloudctl login gcp-terrorgems'"
}
```

### 4. Async/Await Pattern
All operations are properly async for Claude integration:
```python
@pytest.mark.asyncio
async def test_switch_context():
    result = await cloudctl.switch("myorg")
    assert result["account"] is not None
```

## How This Skill is Used

## Important Behaviors

### 1. Multi-Session Safety
Each Claude session can have a different context:
```python
# Session A
await cloudctl.switch("myorg")  # AWS context
execute_terraform()

# Session B (different Claude session)
await cloudctl.switch("gcp-terrorgems")  # GCP context
execute_terraform()  # Uses GCP credentials
```

### 2. Credential Scoping
Credentials are namespace-isolated:
- **AWS**: AWS_ACCESS_KEY_ID, AWS_SECRET_ACCESS_KEY, AWS_SESSION_TOKEN
- **GCP**: GOOGLE_CLOUD_PROJECT, CLOUDSDK_CORE_PROJECT, GOOGLE_OAUTH_ACCESS_TOKEN
- **Azure**: AZURE_SUBSCRIPTION_ID, AZURE_TENANT_ID, ARM_ACCESS_TOKEN

### 3. Non-Interactive Fallbacks
When running in subprocess (Claude):
- GCP: Uses placeholder tokens if interactive re-auth fails
- Azure: Uses cached tokens
- AWS: Delegates to SSO cache (requires prior login)

### 4. Context Persistence
Context is stored in `~/.config/cloudctl/context.json`:
```json
{
  "org": "gcp-terrorgems",
  "account": "asatst-gemini-api-v2",
  "role": "roles/editor",
  "region": "us-central1"
}
```
